               STRUCTURED DATA FORMAT (Recommended)
================================================================

Use structured data when you already have well-organized failure information.

REQUIRED FIELDS:

1. failure_mode (string, 5-500 chars)
   â””â”€ Description of how/why the component could fail
   â””â”€ Example: "Engine fails to start"

2. effect (string, 5-500 chars)
   â””â”€ The consequence or impact of the failure
   â””â”€ Example: "Vehicle cannot be operated"

3. cause (string, 5-500 chars)
   â””â”€ Root cause(s) of the failure
   â””â”€ Example: "Battery dead or starter motor malfunction"

OPTIONAL FIELDS:
â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€â”€
- component (string): The affected component/subsystem
  Example: "Engine Starter System"

- process (string): The process where failure occurs
  Example: "Engine Start Sequence"

- function (string): Function/purpose of the component
  Example: "Initiating engine combustion"

- severity (integer, 1-10): Impact on customer (1=low, 10=critical)
  If not provided, system will calculate automatically

- occurrence (integer, 1-10): Likelihood of occurrence (1=rare, 10=very likely)
  If not provided, system will calculate automatically

- detection (integer, 1-10): Detection difficulty (1=easy, 10=impossible)
  If not provided, system will calculate automatically

- existing_controls (string): Current preventive/detective controls
  Example: "Visual battery inspection"

- recommended_action (string): Action to eliminate/reduce risk
  Example: "Replace battery or repair starter motor"

- responsibility (string): Owner of the action
  Example: "Maintenance Team"

- target_completion_date (string, YYYY-MM-DD format): Deadline
  Example: "2024-03-15"

- additional_notes (string): Extra context or remarks
  Example: "More common in cold weather"

- source (string): Source of failure info
  Valid values: review, complaint, incident_report, customer_feedback, qa_report, warranty_claim, field_report, test_report, other
